## 1. Principios Fundamentales

### 1.1 Propósito y Restricciones
- **Objetivo:** Generar presentaciones académicas de alto nivel para sustentaciones (típicamente 30 minutos).
- **Insumo exclusivo:** Documentación contenida en carpeta `/doctg` (nunca papers externos, datos complementarios o fuentes ajenas).
- **Fidelidad:** Transcribir exactamente nomenclatura, fórmulas y metodología del documento original.
- **Anti-patrón crítico:** No saturar diapositivas con párrafos copiados; priorizar visualización (columnas, tablas, listas).

### 1.2 Stack Tecnológico
```
\documentclass[aspectratio=169, 11pt]{beamer}  % 16:9 widescreen obligatorio
```
- **Motor:** XeLaTeX o LuaLaTeX (flexibles según disponibilidad del sistema).
- **Tipografía:** Usar `fontspec` si se especifican fuentes del sistema; si no, usar defaults de Beamer.
- **Compilación:** `pdflatex -interaction=nonstopmode` como fallback simplificado.

## 3. Estructura y Diseño de Diapositivas

### 3.1 Rango Recomendado
- **17-24 diapositivas** para una presentación de ~30 minutos (1.2-1.5 min por frame).
- **Este es un rango guía:** ajustar siempre según las preferencias y necesidades del usuario. Si especifica otro rango, respetarlo.

### 3.2 Diapositivas Obligatorias
1. Portada (título del proyecto)
2. Tabla de Contenidos (`\tableofcontents`)
3. Contenido temático (metodología + resultados priorizado)
4. Conclusiones
5. Cierre ("Gracias" o similar)

### 3.3 Anti-Patterns — EVALUAR CASO A CASO
| ❌ Generalmente evitar | ✅ Hacer en su lugar |
|-----------|---------|
| Copiar párrafos largos del documento | Sintetizar en viñetas, columnas o tablas |
| Cambiar nomenclatura de variables/funciones | Transcribir exactamente como en la tesis |
| Insertar código extenso sin procesar | Abstraer con bloques matemáticos; código en slides de respaldo |
| Más de 25 diapositivas para 30 minutos | Limitar a 17-24 frames; respetar pacing |
| Ignorar límites de tiempo | Incluir notas `\note[item]{Tiempo: X min}` en cada frame |
| Sin slides de respaldo para preguntas | Agregar `\appendix` con soporte técnico riguroso |

**Nota:** Estos son patrones recomendados, no obligatorios. Si el usuario especifica una excepción (ej. "necesito más de 25 slides"), preguntar antes de aplicar la restricción.

### 3.4 Estructura Recomendada por Sección
```
\section{TemaX}
\input{sections/0X_nombreX.tex}   % 1-3 frames por sección

% Dentro de cada frame:
\begin{frame}[fragile]
  \frametitle{Título Específico}
  
  \begin{block}{Concepto Clave}
    [Contenido sintético]
  \end{block}
  
  \begin{itemize}
    \item Punto principal
    \item Evidencia / ejemplo
  \end{itemize}
  
  \note[item]{Tiempo estimado: X-Y minutos}
\end{frame}
```

## 4. Protocolo Obligatorio para Agentes: Fase 0 Planificación

**ANTES de escribir cualquier código Beamer, ejecutar siempre Fase 0:**

### 4.1 Preguntas de Optimización
1. **Énfasis de contenido:** ¿Qué secciones requieren mayor despliegue? (ej. diseño del modelo vs. experimentos)
2. **Complejidad matemática:** ¿Derivación formal completa o intuición conceptual?
3. **Selección de evidencia:** ¿Cuáles tablas/gráficos de `/doctg` son indispensables?
4. **Duración esperada:** ¿Exactamente 30 min o rango flexible?
5. **Visuales específicas:** ¿Mapas, ecuaciones, gráficos de desempeño prioritarios?

### 4.2 Revisión Holística Pre-Desarrollo
- Verificar paridad de ecuaciones con documento base.
- Evaluar flujo de metodología (bloques lógicos coherentes).
- Identificar riesgos de saturación visual o incompatibilidades de compilador.
- Estimar diapositivas necesarias vs. tiempo disponible.

### 4.3 Resultado de Fase 0
Documento `plan.md` o similar con:
- Objetivos de la sesión
- Estructura de diapositivas propuesta (18-24 frames)
- Checklist de tareas por fase
- Tiempo estimado por bloque

**No proceder al desarrollo sin confirmación del usuario sobre el plan.**

## 5. Gestión de Documentación y Checkpoints

### 5.1 Documentación Dinámica
Mantener dos tipos de documentos en carpeta raíz:

#### **plan.md** (Se elabora al inicio)
- Estructura lógica de diapositivas
- Fases de implementación (Fase 1, 2, 3, ...)
- Checklist de tareas con `[ ]` / `[x]`
- Restricciones críticas (duración, rango de slides)

#### **checkpoints/** (Uno por fase o sesión)
```
checkpoints/
├── plan_fase1_completada.md
├── plan_fase2_completada.md
├── plan_fase3_completada.md
└── plan_fase4_completada.md
```

Estructura base de checkpoint:
```markdown
# Checkpoint: Fase [X] Completada

**Fecha:** YYYY-MM-DD
**Estado:** ✓ / ⏳ / ❌

## Resumen de cambios
- [Listado de frames completados]
- [Gráficos/imágenes incorporadas]
- [Archivos modificados]

## Estadísticas
- Diapositivas: [N]
- Tiempo estimado: [M] minutos
- Tamaño PDF: [S] MB

## Próximos pasos
- [ ] Tarea 1
- [ ] Tarea 2
```

### 5.2 Nomenclatura
- Archivos LaTeX: `sections/NN_tema_descriptivo.tex`
- Gráficos: `figures/nombre_descriptivo.png` (máx 2.0 MB totales)
- Checkpoints: `checkpoints/plan_faseX_completada.md`
- Plan principal: `plan.md` (actualizar dinámicamente)

### 5.3 Registro de Cambios
**Actualizar `plan.md` al cerrar cada tarea:**
```markdown
### Fase X: Descripción
- [x] Tarea 1 (completada)
- [ ] Tarea 2 (pendiente)
- [x] Tarea 3 (completada)
```

**Crear checkpoint solo cuando:**
- Se completa una fase mayor (1, 2, 3, etc.)
- Se alcanza un milestone (PDF compilable, gráficos incorporados, etc.)

## 6. Flujo de Desarrollo Estándar

### Sesión típica (4 fases)

**Fase 1: Esqueleto**
- Crear carpeta `mi_sustentacion/`
- Generar `main.tex` con preámbulo
- Crear 9-10 archivos `sections/` con templates
- Checkpoint: PDF compilable con placeholders

**Fase 2: Contenido Narrativo**
- Llenar secciones con contenido de `/doctg`
- Sintetizar en viñetas, bloques, tablas
- Validar paridad de ecuaciones
- Checkpoint: Contenido completo (aunque pueden faltargráficos)

**Fase 3: Control de Tiempo**
- Analizar duración por frame (ver notas `\note[item]`)
- Expandir/condensar según timing
- Dividir frames si es necesario
- Validar: 17-24 diapositivas, ~30 min
- Checkpoint: Estructura balanceada

**Fase 4: Actualización Dinámica**
- Incorporar gráficos reales de `/doctg`
- Integrar imágenes en frames estratégicas
- Recompilar y validar legibilidad
- Checkpoint: PDF final con gráficos

## 7. Compilación y Validación

### 7.1 Comando de compilación recomendado
```bash
cd mi_sustentacion/
rm -f *.aux *.log *.out *.toc *.nav *.snm *.vrb 2>/dev/null
pdflatex -interaction=nonstopmode main.tex
```

### 7.2 Validación post-compilación
```bash
pdfinfo main.pdf | grep -E "Pages|Title"
```

### 7.3 Criterios de éxito
- ✓ Compila sin errores
- ✓ Número de páginas esperado (17-24)
- ✓ Título visible en metadata
- ✓ Gráficos embebidos (si aplica)

## 8. Notas Especiales por Proyecto

### 8.1 PINN Caribe Oceanográfico (Referencia)
**Este archivo se generó como base de este proyecto específico:**
- Énfasis en ecuaciones de Navier-Stokes y PINNs
- 18 diapositivas finales
- Mapa de estaciones + gráficos de convergencia
- Tiempo: ~32 minutos (2 min sobre límite, aceptable)

**Lecciones aprendidas:**
1. Dividir antecedentes en dos frames (tradicional vs. moderno) → claridad narrativa
2. Agregar frame de formulación matemática → rigor sin sobrecarga
3. Dividir resultados → permite análisis profundo
4. Gráficos tangibles → visualización del problema y convergencia

### 8.2 Adaptabilidad
Estas instrucciones se escribieron para ser generalizables a otros trabajos de grado. **Partes flexibles:**
- Paleta de colores: usar si aplica; simplificar si es necesario
- Tipografía: defaults de Beamer si fontspec no está disponible
- Duración: 30 min es guía; aceptar 28-35 según contexto
- Fases: pueden condensarse (ej. Fase 1+2 en una sesión)
